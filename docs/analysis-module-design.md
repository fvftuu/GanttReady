# 分析模块四Tab设计文档

> 版本：v1.0 | 日期：2026-05-20 | 状态：设计稿

---

## 一、背景与目标

### 现状
当前分析页面有 7 个子 Tab（概况/挣值分析/进度曲线/工期偏差/阶段完成率/资源负荷/资源平衡），全部面向项目经理角色，其他角色（监管方、甲方、总公司）没有专属视图。

### 目标
将分析页面重构为四个主 Tab，分别服务四类用户：

| 编号 | Tab 名称 | 服务角色 | 定位 |
|------|---------|---------|------|
| ④ | 项目总览 | 总公司管理者 | 多项目健康度排雷 |
| ① | 实施分析 | 项目经理 | 日常操作台（现有7Tab） |
| ② | 月度分析 | 监管方/分管领导 | 15分钟月度汇报 |
| ③ | 年度分析 | 甲方/公司高层 | 年度复盘/结项报告 |

---

## 二、页面顶层结构

```
┌── 项目选择器 ─────────────────────────────────────────────┐
│  [下拉选择：所有项目 / 具体项目]                           │
├── ④ 项目总览 │ ① 实施分析 │ ② 月度分析 │ ③ 年度分析 ──┤
│                                                           │
│  （当前 Tab 内容区）                                       │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

**切换逻辑**：
- 选择「所有项目」→ 默认显示 ④ 项目总览，①/②/③ 不可用
- 选择「具体项目」→ 默认显示 ① 实施分析，四个Tab均可切换

## 三、Tab ④ 项目总览

### 服务角色
总公司管理者，手上同时管 5-20 个项目。

### 核心诉求
一眼看穿哪个项目有问题，按优先级处理——红的先管，绿的不管。

### 设计原则
只看例外，不给细节。这不是操作台，这是排雷仪。

### 页面结构

#### 1. 筛选栏
- 时间范围：本季度 / 本年 / 自定义
- 项目分组：全部 / 按类型 / 按负责人

#### 2. 仪表板卡片
| 卡片 | 内容 | 含义 |
|------|------|------|
| 项目总数 | 进行中 / 已完结 | 工作量总览 |
| 健康度分布 | 🟢X个 🟡X个 🔴X个 | 一眼看风险面 |
| 平均 SPI | 数值 + ↑↓箭头 | 整体进度绩效趋势 |
| 平均工期偏差 | 数值（天） | 整体延期程度 |
| 资源冲突项目数 | 数值 | 资源管理压力 |

#### 3. 项目健康度矩阵（表格 — 核心模块）

| 项目名称 | 健康度 | SPI | 工期偏差 | 关键路径变化 | 资源冲突 | 操作 |
|---------|:-----:|:---:|:-------:|:----------:|:-------:|:----:|
| 项目A | 🟢 | 0.98 | +2天 | ±0 | 无 | 查看 |
| 项目B | 🟡 | 0.92 | +8天 | +3天 | 土建工超额 | 查看 |
| 项目C | 🔴 | 0.85 | +25天 | +10天 | 3项冲突 | 查看 |

- 健康度自动判定规则：
  - 🟢 SPI ≥ 0.95 且 工期偏差 ≤ 3天
  - 🟡 SPI ≥ 0.85 且 工期偏差 ≤ 10天
  - 🔴 其余情况
- 点击「查看」跳转到该项目的实施分析（Tab ①）

#### 4. SPI 排名柱状图
- 横轴：项目名称
- 纵轴：SPI 值
- 基准线：SPI=1.0（绿色虚线）
- 低于基准线的柱子标红色

#### 5. 资源冲突跨项目分布
- 列出被多个项目同时使用的共享资源
- 显示：资源名称 / 使用项目数 / 具体项目列表 / 建议

#### 6. 高风险项目列表
- 自动提取健康度为红色的项目
- 显示核心问题摘要

### 后端接口需求
```
GetMultiProjectOverviewAsync() → MultiProjectOverviewResult
  - ProjectHealthItems: List<ProjectHealthItem>
    - ProjectId, ProjectName, HealthStatus, SPI, ScheduleDeviation, 
      CriticalPathChange, ResourceConflicts, Status
  - ResourceConflictCrossProject: List<CrossProjectConflict>
  - AverageSPI, AverageDeviation, HealthDistribution
```

## 四、Tab ① 实施分析

### 服务角色
项目经理，天天用系统排计划、调资源、找问题。

### 核心诉求
钻取数据、定位原因、做出决策。

### 设计原则
保持现有操作习惯，只修痛点，不动筋骨。

### 页面结构

#### 顶部：仪表板卡片（保持现有 7 张）
- 项目工期 / 总任务数 / 关键路径任务数 / 最早完成 / 资源冲突 / SPI / 进度偏差天数
- **新增**：左上角大字项目状态条（🟢 正常 / 🟡 预警 / 🔴 失控）

#### 统计卡片（保持现有 5 张）
- 按时任务 / 提前任务 / 延后任务 / 里程碑节点 / 资源分配项

#### 7 个子 Tab 导航（保持现有）

| 子 Tab | 状态 | 说明 |
|--------|:----:|------|
| 📋 概况 | ✅ 保留 | 关键路径表 + 时差图 + 里程碑 + 资源冲突检测 |
| 💰 挣值分析 | ✅ 保留 | PV/EV/SPI卡片 + S曲线 |
| 📈 进度曲线 | ✅ 保留 | 计划vs实际累计完成率折线图 |
| ⚠ 工期偏差 | ✅ 保留 | 任务偏差表格 + 筛选 |
| 🎯 阶段完成率 | ✅ 保留 | 阶段划分 + 完成率柱状图 |
| 🔧 资源负荷 | ✅ 保留 | 月度负荷矩阵 + 热力图 |
| ⚖ 资源平衡 | ✅ 保留 | 冲突检测 + 调整建议 |

#### 微调改进

| 改动 | 原因 | 工作量 |
|------|------|:------:|
| 顶部加项目状态条（🟢/🟡/🔴） | 先看结论再看证据 | 小 |
| 所有表格加「只看异常」切换开关 | 30个任务只想看有问题的3个 | 小 |
| 删除「资源容量 vs 峰值使用」柱状图 | 不好理解，已被资源负荷Tab覆盖 | 小 |
| SPI 卡片加 6周趋势迷你图 + ↓↑箭头 | 单点数值无意义，趋势才有意义 | 中 |
| 资源冲突表加「建议行动」列 | 不只告诉你有问题，告诉你怎么解决 | 小 |

### 后端接口需求
现有接口基本够用。新增：
```
GetTrendComparisonAsync(int projectId) → TrendComparisonResult
  - CurrentSPI, LastMonthSPI, Trend arrow
  - CurrentCriticalPathLength, LastMonthCriticalPathLength
  - CurrentDeviation, LastMonthDeviation
```
依赖历史快照存储。

## 五、Tab ② 月度分析

### 服务角色
监管方 / 分管领导，每月听一次汇报，15 分钟。

### 核心诉求
不听过程，只看偏差、风险、决策事项。

### 设计原则
一页 A4，前 3 分钟讲完结论，剩下 12 分钟讨论决策。

### 页面结构（A4 打印布局）

#### 第1区：30秒全局仪表盘

```
┌──────────────────────────────────────────────────────────┐
│  月度项目分析报告              2026年5月                  │
│  ─────────────────────────────────────────────────────── │
│                                                          │
│   项目状态：🟡 预警   关键线路：+3天 ↑   SPI: 0.93 ↓    │
│   里程碑达成：4/6 ✅    高风险：2项 🔴                    │
│   预算偏差：-2.3% ✅    资源冲突：1项 ⚠                  │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### 第2区：本月偏差详情
- **进度偏差 TOP3**（表格）
  - 任务名称 / 计划工期 / 实际工期 / 偏差天数 / 原因 / 已采取措施
- **关键路径变化**
  - 文字说明 + 对比：上月关键路径长度 → 本月关键路径长度
  - 新增/消失的关键任务
- **里程碑达成情况**
  - 时间轴展示：✅ 达成标绿，❌ 未达标标红
  - 未达成原因简述

#### 第3区：资源与成本
- **资源冲突 TOP3**
  - 资源名称 / 冲突时段 / 缺口数量 / 建议方案
- **EVM 简化曲线**
  - PV + EV 两条折线（按月聚合数据点）
  - 不展示 AC（甲方视角才需要 AC）

#### 第4区：风险与决策
- **需要领导决策的事项**（不超过 2 项）
  - 事项描述 / 建议方案 / 期望决策时间
- **下月重点关注任务**（不超过 5 项）
  - 任务名称 / 计划完成日期 / 风险提示

### 交互与导出
- 页面采用 A4 宽度布局（210mm 约束）
- 右下角提供「🖨 打印」按钮
- 调用浏览器打印功能，生成 PDF
- CSS 增加 `@media print` 规则优化打印效果

### 后端接口需求
```
GetMonthlyReportAsync(int projectId, int year, int month) → MonthlyReportResult
  - Header: ProjectName, ReportPeriod, HealthStatus
  - Dashboard: SPI, CriticalPathDeviation, MilestoneRate, HighRiskCount, BudgetDeviation, ResourceConflictCount
  - ScheduleDeviations: List<DeviationItem> (TOP3)
    - TaskName, PlannedDuration, ActualDuration, DeviationDays, Reason, Measure
  - CriticalPathChange: CriticalPathComparison
    - LastMonthLength, CurrentLength, NewCriticalTasks, LostCriticalTasks
  - Milestones: List<MilestoneStatus>
  - ResourceConflicts: List<ConflictItem> (TOP3)
  - Decisions: List<DecisionItem> (≤2)
  - NextMonthFocus: List<FocusItem> (≤5)
  - EVMData: MonthlyEVMData (PV/EV by month)
```

## 六、Tab ③ 年度分析

### 服务角色
甲方 / 公司高层，一年看一次或项目结束时看。

### 核心诉求
钱花得值不值？结果好不好？下次怎么改进？

### 设计原则
这是一份可存档的「项目病历」——哪里做得好、哪里出了问题的完整记录。

### 页面结构（3-5 页，PDF导出）

#### 第1页：项目总览卡

```
┌──────────────────────────────────────────────────────────┐
│  项目名称：综合基础设施建设项目                            │
│  ─────────────────────────────────────────────────────── │
│                                                          │
│  工期：计划 360天 → 实际 385天（+25天，+6.9%）⚠          │
│  成本：计划 ¥2,000万 → 实际 ¥2,150万（+7.5%）⚠          │
│  里程碑达成：10/12（83%）⚠                                │
│  安全：0 事故 ✅                                          │
│  质量：验收通过率 100% ✅                                  │
│                                                          │
│  整体评价：🟡 基本达标                                    │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

评价规则：
- 🟢 达标：工期偏差 ≤ 5% 且 成本偏差 ≤ 5% 且 里程碑达成率 ≥ 90%
- 🟡 基本达标：工期偏差 ≤ 15% 且 成本偏差 ≤ 10%
- 🔴 未达标：其余情况

#### 第2页：核心指标对比

| 指标 | 计划值 | 实际值 | 偏差 | 评价 |
|------|--------|--------|------|:----:|
| 总工期 | 360天 | 385天 | +25天（+6.9%） | ⚠ |
| 总成本 | ¥2,000万 | ¥2,150万 | +150万（+7.5%） | ⚠ |
| 关键路径变更次数 | — | 3次 | — | ⚠ |
| 里程碑达成率 | 100% | 83% | -17% | ⚠ |
| 平均资源利用率 | 85% | 78% | -7% | ✅ |
| 安全事故 | 0 | 0 | — | ✅ |

#### 第3页：阶段回顾

按项目阶段展示（准备 / 基础 / 主体 / 装修 / 收尾）：

| 阶段 | 计划工期 | 实际工期 | 偏差 | 主要原因 | 经验教训 |
|------|---------|---------|:----:|---------|---------|
| 施工准备 | 30天 | 28天 | -2天 | 审批流程优化 | ⋯ |
| 基础工程 | 60天 | 75天 | +15天 | 地质条件变化 | ⋯ |
| 主体结构 | 120天 | 130天 | +10天 | 材料供应延迟 | ⋯ |
| 机电安装 | 90天 | 92天 | +2天 | — | ⋯ |
| 装修收尾 | 60天 | 60天 | 0天 | — | ⋯ |

#### 第4页：风险与问题

- **年度风险热力图**：12 个月的色块矩阵，颜色代表当月风险等级
- **主要风险应对效果评估**：
  | 风险 | 发生月份 | 影响程度 | 应对措施 | 效果 |
  |------|---------|:-------:|---------|:----:|
  | ⋯ | ⋯ | ⋯ | ⋯ | ⋯ |
- **遗留问题清单**：
  | 问题 | 状态 | 责任人 | 计划解决时间 |
  |------|:----:|:-----:|:----------:|

#### 第5页：改进建议

- **管理流程改进**（2-3 条）
  - 如：加强材料供应商预审、增加地质勘察阶段
- **技术/工具改进**（2-3 条）
  - 如：引入 BIM 碰撞检测、使用自动进度采集工具
- **下一年/下一个项目的输入**
  - 总结性建议

### 交互与导出
- 页面分页展示（5 页）
- 提供「📄 导出 PDF」按钮
- 专业排版，适合存档

### 后端接口需求
```
GetAnnualReportAsync(int projectId, int year) → AnnualReportResult
  - Header: ProjectName, PlannedDuration, ActualDuration, 
    PlannedCost, ActualCost, MilestoneTotal, MilestoneAchieved, 
    SafetyIncidents, QualityPassRate, OverallRating
  - KpiComparison: List<KpiItem>
    - KpiName, PlannedValue, ActualValue, Deviation, Rating
  - PhaseReviews: List<PhaseReview>
    - PhaseName, PlannedDuration, ActualDuration, Deviation, 
      MainReason, Lessons (3 items)
  - RiskReview: AnnualRiskReview
    - MonthlyRiskLevels: List<int> (12 months, 1-3)
    - RiskResponses: List<RiskResponse>
    - RemainingIssues: List<IssueItem>
  - Improvements: List<ImprovementItem> (2-3 per category)
```

## 七、四个Tab的协作逻辑

```
④ 项目总览             ① 实施分析
（扫射：哪个项目有问题？） （钻取：问题出在哪？）
        │                      │
        │ 点项目名跳转          │
        └──────────┬────────────┘
                   │
        ② 月度分析（本月汇报材料）
        ③ 年度分析（年度总结文档）
```

1. **④ → ①**：总览看到项目B是红色的 → 点进去进①看具体哪里有问题
2. **① → ②**：项目经理在①中发现的问题 → 月底自动聚合生成②给领导汇报
3. **① → ③**：年底自动聚合①中的全年数据 → 生成③存档

**三个层次不重复**：
- ④ 是排雷仪 — 看一眼知道哪里炸了
- ① 是手术台 — 工具齐全，可以动手
- ② 是体检报告 — 每月一次，给领导看
- ③ 是病历档案 — 项目结束后存档

## 十、附录：健康度判定规则

### 项目健康度（用于 ④ 项目总览）

| 等级 | SPI | 工期偏差 | 资源冲突 |
|:----:|:---:|:--------:|:--------:|
| 🟢 正常 | ≥ 0.95 | ≤ 3天 | 无 |
| 🟡 预警 | ≥ 0.85 | ≤ 10天 | ≤ 2项 |
| 🔴 失控 | < 0.85 | > 10天 | > 2项 |

> 取最差维度为最终等级。例如 SPI=0.96（绿）但偏差=15天（红），整体为 🔴。

### 年度评价（用于 ③ 年度分析）

| 等级 | 工期偏差 | 成本偏差 | 里程碑达成率 |
|:----:|:--------:|:--------:|:-----------:|
| 🟢 达标 | ≤ 5% | ≤ 5% | ≥ 90% |
| 🟡 基本达标 | ≤ 15% | ≤ 10% | ≥ 70% |
| 🔴 未达标 | > 15% | > 10% | < 70% |

> 取最差维度为最终等级。
