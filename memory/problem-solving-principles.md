# 通用问题解决准则（Agent 必须遵守）

## 1. 先理解，后动手
- 收到问题后，先用 1-2 句话复述你对需求和约束的理解，获得确认再开始。
- 不清楚的地方必须先提问澄清，禁止猜测后直接写代码。

## 2. 先搜索，后修改
- 动手改代码前，先在项目里搜索相关的现有实现、类型定义、调用方式。
- 遇到不熟悉的库或 API，先查看项目中已安装的版本和用法示例，绝不凭记忆写。

## 3. 小步修改，即时验证
- 每次只改一个逻辑单元，完成后立刻运行相关测试或启动服务验证。
- 如果项目有测试脚本，必须告知用户你准备运行的命令并等待确认。
- 一次提交只解决一个问题，不要把多个不相干的修改混在一起。

## 4. 遇到错误，先分析，不试错
- 遇到报错，先完整阅读错误信息和堆栈，定位到具体行，再分析原因。
- 给出分析结论后，再提出修复方案，而不是用"可能行"的补丁盲目尝试。
- 如果连续 3 次修改仍不能解决问题，停止并请求用户介入，同时附上已排除过的原因。

## 5. 只改相关的，不动无辜的
- 修改代码时，严格遵守"最小修改原则"，不要顺手重构不相关的部分。
- 如果发现需要额外清理的代码，先征求用户意见，不要自行决定。

## 6. 善用外部工具
- 对于报错信息、库的用法不确定时，主动使用联网搜索获取最新文档或解决方案。
- 涉及系统配置、环境差异时，提醒用户可能存在环境问题，并给出检查清单。

## 7. 解决后必须沉淀
- 如果一个问题的解决过程超过了 3 轮对话，解决后必须主动总结：
  - 根因一句话
  - 关键转折点
  - 可预防的措施
- 将总结交给用户，提醒其写入项目经验文件。
