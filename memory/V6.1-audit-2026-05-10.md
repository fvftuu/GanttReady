# NetPlan V6.1 审计报告

**日期**: 2026-05-10  
**审计范围**: Network.razor (1141行) + netplan.js (1752行) + site.css  
**审计方法**: 逐行代码审查 + 数据流跟踪 + 边界情况分析

---

## 总体评估

NetPlan V6.1 时标双代号网络计划图功能**主体实现质量较高**。核心数据流（C# BuildDataInternal → JSON → JS renderNetwork → buildNetworkSvg → SVG）完整且正确。JS↔C# Blazor 互调接口（JSInvokable / JSInvokeVoidAsync）设计清晰，没有发现阻塞级（P0）缺陷。

主要问题集中在：**参数化配置与实际使用脱节**（如层高修改后垂直微调失效）、**部分视觉元素的精度和完整度不足**（波形线分段数固定、图例少一项、进度曲线与标题重叠）、以及**少量死代码和未连通功能**（rowLabels 永不传值、exportNetworkSVG 未调用）。整体评估为 **"功能通过，需修复9项中/低级问题"**。

## P1 严重缺陷（3项）

### P1-1: 垂直微调 (A1) 在自定义层高下完全失效

**问题**: 节点沿着 Y 轴的拖动量化使用硬编码 `LAYER_H = 60`，但当图属性面板中 `graphLayerHeight` 被改为非 60 的值时，实际 SVG 坐标已被缩放，导致：
- 节点 snap 到错误的网格
- `_netExtraSpacing`（插入/删除空行）也使用硬编码 `60` 计算层索引
- 空行插入后的 `evt.y += lh` 与之前的层索引计算不一致

**代码位置**:
- `netplan.js` 全局 `document.addEventListener('mousemove', ...)` 中: `var dy = _netNodeDrag.offY + Math.round(rawDy / LAYER_H) * LAYER_H;`（LAYER_H 硬编码60）
- `netplan.js` `_netInsertBlankRow()`: `var layerIdx = Math.round((evt.y - 100) / 60);`（60硬编码）
- 对比 `renderNetwork()` 中: `Object.keys(layout.events).forEach(function(eid) { layout.events[eid].y = 100 + (layout.events[eid].y - 100) * layerScale; });`（层高缩放在渲染时应用，但权重事件层索引仍然基于60）

**严重性**: 核心编辑功能（纵向微调、空行插入）在用户改动层高后产生错误行为。

---

### P1-2: 进度曲线 (E1) 与标题文本重叠

**问题**: 进度曲线的绘制区域为 `curveYMax = 56`（紧接时间标尺底部）到 `curveYMin = 88`，但标题文本绘制在 `y=80` 和 `y=97`。进度曲线的透明蓝色填充与标题文本直接重叠，影响可读性。

**代码位置**: `netplan.js` `buildNetworkSvg()` — 进度曲线区域:
```js
var curveYMin = 88; // 曲线区域底部
var curveYMax = 56; // 曲线区域顶部（紧挨标尺底部52）
```
标题文本在之后绘制:
```js
parts.push('<text x="10" y="80" ...>' + (p.projectName || '') + '</text>');
parts.push('<text x="10" y="97" ...>' + subtitle + '</text>');
```

**严重性**: 当进度曲线开启时，标题被蓝色半透明填充遮挡或干扰。

---

### P1-3: 自由时差波形线 (C1c) 分段数固定导致视觉效果差

**问题**: 波形线（FF>0 金色波形）仅使用 10 段锯齿 (`for (var s = 0; s < 10; s++)`)，无论 FF 跨度多大。当 FF=20 天时，10 段锯齿要覆盖 20 个日期的宽度，导致波形极度稀疏、不连贯，失去波浪视觉效果。

**代码位置**: `netplan.js` `buildNetworkSvg()`:
```js
for (var s = 0; s < 10; s++) {
    var t = s / 10;
    wp += ' L' + (ex + (fex - ex) * t) + ' ' + (ey + (s % 2 ? -3 : 3));
}
```

**修复建议**: 采样数应基于波形线实际宽度，至少每宽度≈8px 一个采样点（例如 `var segments = Math.max(8, Math.ceil(act.ff * p.dayWidth / 8))`）。

## P2 一般缺陷（6项）

### P2-1: rowLabels (E5) 永远为空数组，功能未连通

**问题**: C# 侧在 `BuildDataInternal` 和 `RerenderGraph` 中始终传递 `rowLabels = new object[] { }`。JS 渲染逻辑已支持对象格式的行标注，但没有任何 UI 或数据流来填充行标签。这是一个"半实现"功能。

**代码位置**: 
- `Network.razor` 第 622 行、983 行: `rowLabels = new object[] { },`
- `netplan.js` `buildNetworkSvg()` 中对 `p.rowLabels` 的渲染代码已就绪

**严重性**: 标注功能属于 V6.1 清单 E5，但用户无法使用。

---

### P2-2: 图例仅显示 3 项，V6.1 清单要求 4 项

**问题**: 底部图例只绘制了"关键线路"、"非关键工作"、"虚工作" 3 项，但 V6.1 清单明确要求"底部图例 4 项"。缺少第 4 项。

**代码位置**: `netplan.js` `buildNetworkSvg()` 图例区域:
```js
[
    { label: '关键线路', color: criticalColor, ... },
    { label: '非关键工作', color: normalColor, ... },
    { label: '虚工作', color: dummyColor, ... }
].forEach(...)
```

**严重性**: 功能规格不满足。

---

### P2-3: ZoomToFit 计算未考虑 MARGIN_LEFT (80px) 偏移

**问题**: ZoomToFit 的可用宽度计算为 `(bodyWidth - 100) / totalDays`，但 SVG 的实际水平占用为 `MARGIN_LEFT(80) + totalDays * dayWidth + 100`。ZoomToFit 多算了约 80px 的可用空间，导致缩放后右侧被裁剪。

**代码位置**: `Network.razor` `ZoomToFitAsync()`:
```csharp
var targetDayW = (bodyWidth - 100) / Math.Max(1, totalDays);
```
应改为:
```csharp
var targetDayW = (bodyWidth - 80 - 100) / Math.Max(1, totalDays);
```

---

### P2-4: 进度曲线填充路径起始点有 dw/2 偏移

**问题**: 填充路径从 `MARGIN_LEFT + 0 * dw`（即 80px）开始，但第一个曲线数据点在 `MARGIN_LEFT + 0 * dw + dw/2`（即 80 + dw/2）。填充底边与第一个数据点之间存在 dw/2 的水平缺口。

**代码位置**: `netplan.js` `buildNetworkSvg()` 进度曲线填充:
```js
var fillPath = 'M' + (MARGIN_LEFT + 0 * dw) + ',' + curveYMin + ' L';
```
应改为:
```js
var firstX = MARGIN_LEFT + 0 * dw + dw / 2;
var lastX = MARGIN_LEFT + (td - 1) * dw + dw / 2;
var fillPath = 'M' + firstX + ',' + curveYMin + ' L';
```

---

### P2-5: `exportNetworkSVG` 函数死代码

**问题**: `netplan.js` 中定义了 `window.exportNetworkSVG` 函数，但 C# `ExportSVG()` 直接调用 `downloadSVG`，从未使用 `exportNetworkSVG`。该函数是未使用的死代码。

**代码位置**: `netplan.js`（约第 1600 行）

---

### P2-6: 进度曲线中 dayCount=0 时 avgPct=0 导致填充到 yMin

**问题**: 当某天没有任何任务活跃时，`avgPct = 0`，曲线数据点位于 `curveYMin`（88px）。虽然不被认为是严重Bug，但会导致曲线在无任务的天数突然"掉到底部"，影响视觉连贯性。更合理的做法是跳过无任务的天数或插值。

## P3 体验优化（6项）

### P3-1: CSS 类 `label-panel` 和 `header-extra` 未在 site.css 中定义

**问题**: Network.razor 使用了 `class="label-panel"` 和 `class="header-extra"`，但 site.css 中没有对应的样式规则。这两个元素完全依赖内联 style 属性工作，CSS 类名是死引用。

**位置**: Network.razor 第 37 行、第 69 行

---

### P3-2: 1080p 视口自适应缩放因子 0.7 过于激进

**问题**: 在 `<1400px` 视口下，dayWidth 降至原始的 0.7 倍。默认 40px → 28px。对于 1080p 分辨率（1920×1080），该条件不触发，但对于 1366×768 等分辨率，28px 的日宽可能使日期数字拥挤。

**代码位置**: `netplan.js` `renderNetwork()`:
```js
if (mode !== 'logic' && vw < 1400 && dayWidth > 12) {
    dayWidth = Math.max(8, dayWidth * 0.7);
}
```

**建议**: 考虑使用 `Math.min(dayWidth, Math.max(8, vw / totalDays * 0.85))` 的浮动缩减策略。

---

### P3-3: C# 序列化时 `eventNum` 字段发送但 JS 从未使用

**问题**: `BuildDataInternal` 中每个任务携带递增的 `eventNum`（行 934），但 JS 通过拓扑排序独立计算事件编号，完全不使用该字段。这是带宽浪费（对大量任务影响可忽略，但仍为代码异味）。

**代码位置**: `Network.razor` 第 934 行:
```csharp
.Append("\"eventNum\":").Append(eventNum++).Append(",")
```

---

### P3-4: A5 线颜色/线宽参数在 C# 硬编码，无 UI 调节入口

**问题**: A5 功能要求"关键/非关键/虚线 颜色和宽度参数化"，但 `criticalColor`, `normalColor`, `dummyColor`, `criticalWidth`, `normalWidth` 在 C# `optionsJson` 中硬编码为固定值。图属性面板（A3）没有提供颜色/宽度调节控件。

**代码位置**: `Network.razor`（`optionsJson` 序列化）
```csharp
criticalColor = "#e63946",
normalColor = "#333",
dummyColor = "#aaa",
criticalWidth = 3,
normalWidth = 1.5
```

---

### P3-5: `_netInsertBlankRow` 使用硬编码 60 计算层索引

**问题**: 插入/删除空行函数使用 `Math.round((evt.y - 100) / 60)` 判断事件所在层，但实际层高可能是 `layerHeight`（来自图属性面板）。虽然初始渲染的层高就是 60（`MARGIN_TOP + layerIndex * LAYER_HEIGHT`），但额外空行后 y 坐标改变，再更改图属性层高会导致层索引混乱。

**代码位置**: `netplan.js` `_netInsertBlankRow()` 和 `_netDeleteBlankRow()`

---

### P3-6: 轮询器 300ms 间隔偏激进

**问题**: `startNetworkPoller` 每 300ms 检查 `netplan-render-token` 的值变化。在页面空闲时仍有规律地触发定时器。建议降低至 500-800ms 以减少 CPU 开销，或者改为 MutationObserver 监听 `#netplan-render-token` 的属性变化。

**代码位置**: `netplan.js` 底部自执行函数:
```js
setInterval(function() { ... }, 300);
```

## 数据流完整性

```
C# BuildDataInternal()
  ↓ 原始数据从 DB (ProjectService)
  ↓ StringBuilder 构建 JSON (tasks + relations)
  ↓ hidden input #netplan-data, #netplan-options, #netplan-render-token
  ↓ renderToken++
  ↓ StateHasChanged()
  ↓ Blazor 更新 DOM
  ↓ 轮询器每隔 300ms 检测 token 变化
JS startNetworkPoller()
  ↓ 检测 netplan-render-token 值变化
  ↓ window.renderNetwork(dataEl.value, JSON.parse(optsEl.value))
  ↓ calculateTimeParams() → events + activities + adjacencies
  ↓ calculateVerticalLayout() → Y coordinates, layer assignment
  ↓ X coordinates (time-based or logic order)
  ↓ 应用 A2 extraSpacing, A3 layerHeight scale
  ↓ 恢复 _netEventOffsets
  ↓ buildNetworkSvg() + ce.innerHTML = SVG string
```

**数据流完整性评估: ✅ 完整**。JSON 格式统一，JS 解析兜底完整（`JSON.parse` + `try/catch`），C# 端有 `skipLoading` 优化路径避免不必要 loading。

## 性能分析

### 重复渲染
- **轮询器 300ms**: 最主要的性能开销点。在页面无变化时，每秒检查 3.3 次 DOM 值。建议提升到 500-800ms。
- `OnAfterRenderAsync` 每次渲染都调用 `setNetworkDotNet`（带 try/catch 保护），但 DotNetObjectReference 创建仅一次。

### 事件绑定
- SVG 的 `ondblclick`、`mousedown` 每次重渲染被替换（因 innerHTML 替换旧 SVG），旧监听自动回收 ✅
- 全局事件（`document.addEventListener`）通过 `_netDragBound` 和 `_netDragSetup` 守卫，只绑定一次 ✅
- 画布拖拽通过 `_panBound` 守卫只绑定一次 ✅

### 大循环
- `calculateTimeParams()`: O(E + V) 拓扑排序
- `calculateVerticalLayout()`: O(E + V) BFS
- `buildNetworkSvg()`: O(V + A + R) — 对 200+ 事件、500+ 任务仍可接受
- SVG 字符串拼接使用数组 `parts.push()` + `.join('')`，性能良好 ✅

### CSS 动画风险
- `#cy svg { transition: transform 0.3s ease; }` — 若频繁更新 innerHTML 替换 SVG，transition 永远无法完成，属性无用但无负面影响。

### 边界情况
- **空项目**: `graphJson = null` → 显示空状态 🕸️ ✅
- **单任务**: ES≈EF 时源和目标事件相同（`T0`），`addEdge` 函数有 `if (f === t) return;` 自环保护 ✅
- **大量任务**: 无 N+1 查询，无内存泄露 ✅
- **无关系**: 每个任务独立，无虚工作生成 ✅
- **循环依赖**: 拓扑排序未覆盖事件通过 `sortedEvents` 兜底分配 ✅
- **前锋线超出范围**: 检查 `progressOffset < 0` 和 `>= td` ✅
- **Async void**: 检查发现 `OnPlanDatesChanged` 等是 `void` 而非 `async void`，调用后 `editingTask` 可能为空。但它们是事件回调，非异步执行不会触发空引用异常。
