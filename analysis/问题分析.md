# NetPlan 项目问题分析报告
> 生成时间：2026-05-14 21:16

基于项目文件、会话记录（5月8日-14日）、源代码结构和 V6.1 功能对标清单进行的综合诊断。

---

## 一、架构层面问题

### 1.1 front-end JS 过大，无模块化
- `netplan.js`: **114KB**，单体文件，包含网络图全部渲染逻辑、交互、计算
- 无 TypeScript、无 ES Module、无自动格式检查
- 难以维护、测试、多人协作

### 1.2 Blazor Server 架构限制
- 项目使用 **Blazor Server**，每个交互需 SignalR 连接
- 导致网络图拖拽、缩放等高频交互依赖 JS 端处理，C# 和 JS 之间存在双重重叠的业务逻辑（如分层计算）
- JSInvokable 方法过多，C#/JS 调用栈容易出错

### 1.3 缺乏单元测试覆盖
- `NetPlan.Server.Tests` 目录存在但未看到实际测试用例
- 核心逻辑（ScheduleEngine、分层算法、事件合并）无自动化测试覆盖
- 修改后靠手动刷新验证，回归成本高

### 1.4 数据库使用 SQLite（netplan.db）
- 当前 `netplan.db` 文件 **114KB** 但非版本控制
- 无迁移策略、无数据备份机制
- 生产环境不适用

## 二、功能缺陷（基于 V6.1 对标清单）

### 2.1 前锋线 —— 最多问题
| 问题 | 严重度 | 详情 |
|------|--------|------|
| 网络图前锋线是竖直线，非折线 | P0 | 行业标准定义：折线连接各工作进度点。当前实现是单条竖直线（检查日期标线），形似神不似 |
| 甘特图前锋线用实线非点划线 | P1 | 行业规范要求点划线 `stroke-dasharray` |
| 甘特图前锋线无检查日期标注 | P1 | 只有折线+圆点，缺少日期标签 |
| 两图前锋线行为和外观不一致 | P1 | 网络图竖直线（可拖），甘特图折线（不可拖），用户困惑 |
| 无超前/拖后数值标注 | P2 | 节点颜色隐式表达了偏差，但无 `+2d`/`-3d` 数值 |

### 2.2 网络图（Network）
| 问题 | 严重度 | 详情 |
|------|--------|------|
| 节点拖拽在缩放模式下的箭线跟随 | P1 | 5月8日已修复但未确认效果，缩放坐标转换仍可能有问题 |
| 节点分层有空洞 | P1 | 事件合并后同层共享节点，导致部分层出现空位，视觉效果不均匀 |
| 从节点拖出创建任务（C1d） | P2 | 功能已标记但未做 |
| 纵向微调节点位置（A1） | P2 | Y 偏移功能存在但用户体验待优化 |
| 底部镜像标尺与图例重叠 | P2 | canvasH 计算含余量但未验证边界情况 |
| 虚工作渲染可能误判 | P2 | 过桥法绘制在复杂关系中可能出现误判 |

### 2.3 甘特图（Gantt）
| 问题 | 严重度 | 详情 |
|------|--------|------|
| 时间标尺三档模式（标准/模式1/模式2） | P2 | 两图标尺对齐未充分验证 |
| 甘特图和网络图前端数据源不统一 | P2 | 前锋线检查日期 JS 用 `_progressCheckDate`，甘特图用当前日期 |
| 甘特表头可能不匹配显示内容 | P2 | 列宽/字段映射在高缩放倍数下可能错位 |

### 2.4 绘制规范合规
| 问题 | 严重度 | 详情 |
|------|--------|------|
| 箭头方向全量验证 | P2 | 仅实现了基础合规，未对整图做方向一致性校验 |
| 母线法绘制平滑度 | P3 | 母线条目在多分支情况下平滑性不足 |
| 过桥法误判优化 | P3 | 复杂交叉线情形 |

## 三、代码质量问题

### 3.1 冗余文件和遗留文件
- `docs/` 和 `docs_utf8/` 两个目录内容重复（编码不同）
- 多个 `.bak`/`.backup` 文件散落在各目录
- 多个 start\* 脚本冗余（start_server.bat, start_server.ps1, start_server2.ps1, start_service.ps1, run_server.bat, run.bat, run_build.bat 等）

### 3.2 JS 代码质量
- 全局变量泛滥：`_netDotNet`, `_netLayout`, `_progressCheckDate`, `_netDayWidth` 等全在全局作用域
- 无严格模式（`"use strict"`）
- 函数体过大：`buildNetworkSvg` 单个函数数千行
- 逻辑分支深度过深：`if/else` 嵌套 5-6 层

### 3.3 C# 代码质量
- `Network.razor`（57KB）和 `Gantt.razor`（99KB）单个文件过大
- 页面逻辑和数据访问混合在同一文件中
- Blazor `@code` 块中的 JSInterop 方法组织混乱

### 3.4 git 管理不规范
- 5月10日记录：修改后未及时 commit，丢失部分修改
- 无分支策略、无 CHANGELOG
- 关键修改（如事件合并架构变动）无架构文档更新

## 四、开发运维问题

### 4.1 构建流程
- 使用 `batch` + `ps1` 混合脚本启动，缺少统一入口
- 无构建流水线（CI/CD）
- 依赖包管理（NuGet）前期有配置问题，已修复但痕迹仍在

### 4.2 运行时
- 运行在 `localhost:5000`，无 HTTPS 配置
- Blazor Server 需要 SignalR，部署前需考虑 WebSocket 支持
- 缺少错误日志和监控

### 4.3 文档
- 核心架构文档（`NetPlan架构文档.utf8.md`）最后一次更新 5月7日，未反映事件合并重构
- 功能清单和实际状态有偏差
- 无 API 文档

## 六、改进优先级建议

### 第一优先：P0 修复 + P1 关键体验
1. 网络图前锋线改造为折线（P0）

3. 清理冗余文件
4. 建立 git commit 习惯和 CHANGELOG

### 第三优先：体验打磨
1. 节点分层空洞优化
2. 纵向微调 UX 改进
3. 箭头方向全量验证
4. C1d 从节点拖出创建任务
