## 修复甘特图总工期计算错误

### 问题描述
用户反馈：当项目工期从未来的时间开始时，总工期似乎加上了空白日期，导致总工期增多。

### 根本原因分析
在 `Gantt.razor` 第477行，总工期计算使用了 `projectStartDate`（默认值为 `DateTime.Today`）。
当项目开始日期设在未来时，如果 `projectStartDate` 没有正确更新，就会用今天作为起始日期，
导致总工期包含了从今天到项目未来开始日期之间的"空白日期"。

### 修复方案
**修改文件**：`/i/NetPlan/src/NetPlan.Server/Pages/Project/Gantt.razor`

**修改内容**（第477行）：
```razor
<!-- 修改前 -->
<span class="status-item">📅 总工期：@(tasks.Any() ? (tasks.Max(t => t.PlanEndDate) - projectStartDate).Days : 0) 天</span>

<!-- 修改后 -->
<span class="status-item">📅 总工期：@(tasks.Any() ? (tasks.Max(t => t.PlanEndDate) - tasks.Min(t => t.PlanStartDate)).Days + 1 : 0) 天</span>
```

**修复说明**：
1. 使用 `tasks.Min(t => t.PlanStartDate)` 替代 `projectStartDate`
2. 总工期现在根据实际任务的最早开始日期和最晚结束日期计算
3. 不再依赖可能不准确的 `projectStartDate` 变量
4. 加1是因为工期计算需要包含开始和结束日期

## 重构时标双代号网络图（vertical layout + 规范绘图）

### 用户需求（严格按照四步执行）
1. **第一步**：计算所有时间参数（ES/EF/LS/LF/TF/FF）— 已在 `ScheduleEngine` 完成
2. **第二步**：垂直分层布局（关键线路在上，非关键工作向下）
3. **第三步**：绘制规范网络图（时标、箭线、波形线、关键线路）
4. **第四步**：添加图例和说明

### 完成的工作

#### 1. 修改 `netplan.js`（核心重构）
- **新的垂直分层布局算法** `calculateVerticalLayout()`：
  - 关键线路工作（TF=0）放在第0层（最上层）
  - 非关键工作按与关键线路距离向下分层
  - 每层内按ES排序，X坐标基于时标（ES × 2 + 边距）
  - 检查同层节点重叠，自动调整

- **规范绘图**：
  - 节点：圆形，直径1.5cm（NODE_R=28，直径56px）
  - 实箭线：水平段长度 = 持续时间 × dayWidth，线宽1.5磅
  - 虚箭线：垂直绘制，线宽1磅，虚线样式([6,4])
  - 波形线：黄色（#FFD700），水平段长度 = 自由时差(FF) × dayWidth
  - 关键线路：红色粗线（3磅），关键节点红色填充
  - 字体：宋体（SimSun）10号，文字居中对齐

- **图例和说明**：
  - 右下角图例：实箭线=工作，虚箭线=逻辑关系，波形线=自由时差，红色=关键线路
  - 左上角总工期标注：总工期=XX天
  - 规范标注：符合 JGJ/T 121-2015

#### 2. 修改 `Network.razor`（数据传递）
- 添加 `ff`（自由时差）字段传递到前端
- 确保 `ff` 字段在 JSON 数据中正确序列化

#### 3. 修复的问题
- 波形线判断从 `tf`（总时差）改为 `ff`（自由时差）
- 箭线绘制从节点边缘出发，而非节点中心
- 跨层箭线改为先水平再垂直的L形
- 文字标注防止重叠（超长文本截断）

### 技术要点
- **时标网络图规范**：节点中心对准时间刻度，箭线水平段长度严格按持续时间比例
- **分层算法**：基于关键路径距离，关键任务层=0，非关键任务层=min(后继关键任务层)+1
- **避免重叠**：同层节点按ES排序，检查X轴间距≥MIN_X_GAP(60px)
