# 甘特图总工期计算问题修复

## 问题描述
用户反馈：当项目工期从未来的时间开始时，总工期似乎加上了空白日期，导致总工期增多。

## 根本原因分析
在 `Gantt.razor` 文件的第477行，总工期的计算方式为：
```razor
(tasks.Max(t => t.PlanEndDate) - projectStartDate).Days
```

**问题**：
- `projectStartDate` 默认值为 `DateTime.Today`（第781行）
- 当项目开始日期设在未来时，如果 `projectStartDate` 没有正确更新，就会用今天作为起始日期
- 这导致总工期包含了从今天到项目未来开始日期之间的"空白日期"

## 修复方案
**修改文件**：`/i/NetPlan/src/NetPlan.Server/Pages/Project/Gantt.razor`

**修改内容**（第477行）：
```razor
<!-- 修改前 -->
<span class="status-item">📅 总工期：@(tasks.Any() ? (tasks.Max(t => t.PlanEndDate) - projectStartDate).Days : 0) 天</span>

<!-- 修改后 -->
<span class="status-item">📅 总工期：@(tasks.Any() ? (tasks.Max(t => t.PlanEndDate) - tasks.Min(t => t.PlanStartDate)).Days + 1 : 0) 天</span>
```

**修复说明**：
1. 使用 `tasks.Min(t => t.PlanStartDate)` 替代 `projectStartDate`
2. 这样总工期将根据实际任务的最早开始日期和最晚结束日期计算
3. 不再依赖可能不准确的 `projectStartDate` 变量
4. 加1是因为工期计算需要包含开始和结束日期（例如：1月1日到1月1日是1天，不是0天）

## 当前遇到的问题
**无法构建验证修复**：
- 系统级 NuGet 配置问题导致 `dotnet restore` 失败
- 错误信息：`Value cannot be null. (Parameter 'path1')`
- 错误位置：`NuGet.Configuration.XPlatMachineWideSetting..ctor()`

**已尝试的解决方案**（均未成功）：
1. 清理并重新还原 NuGet 包
2. 修改 `global.json` 强制使用 .NET 8 SDK
3. 重命名项目级和用户级 `NuGet.Config` 文件
4. 设置 `NUGET_PACKAGES` 环境变量
5. 使用 `--packages` 参数指定包目录
6. 下载并使用 `nuget.exe` CLI 工具
7. 创建简化的 `NuGet.Config` 文件
8. 设置 `NUGET_XPLAT_DISABLEMACHINEWIDESETTINGS` 环境变量

**问题根源**：
- NuGet 在初始化 machine-wide 设置时，尝试获取系统文件夹路径（如 `ProgramFilesX86`）
- 某个路径返回 null，导致 `Path.Combine` 失败
- 这是一个系统级问题，可能需要修复 .NET SDK 安装或重置系统环境变量

## 建议用户执行的操作
1. **在 Visual Studio 中打开项目并构建**（VS 可能使用不同的 NuGet 还原机制）
2. **修复 .NET SDK 安装**：
   - 卸载并重新安装 .NET 8 SDK
   - 或使用 Visual Studio Installer 修复 .NET 开发工具
3. **手动还原 NuGet 包**：
   - 在项目目录中运行：`dotnet restore --force-evaluate`
   - 或删除 `obj` 和 `bin` 文件夹后重新构建
4. **检查系统环境变量**：
   - 确保 `ProgramFiles`、`ProgramFiles(x86)`、`UserProfile` 等环境变量已正确设置

## 修复验证步骤
构建成功后，用户应验证：
1. 启动应用程序
2. 打开甘特图页面
3. 选择一个开始日期在未来的项目
4. 检查底部状态栏中的"总工期"数值
5. 验证总工期 = (最晚任务结束日期 - 最早任务开始日期) + 1
